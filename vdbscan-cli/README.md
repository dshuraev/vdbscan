# vdbscan CLI

Command-line interface for the `vdbscan` point-cloud clustering library. Reads a point cloud, runs DBSCAN, and writes a labelled output file. No algorithm logic lives here — only format codec and argument handling.

## Usage

```
vdbscan -i <input> -o <output> --epsilon <ε> --min-pts <n> [--input-format <fmt>]
```

### Arguments

| Argument | Short | Required | Description |
| --- | --- | --- | --- |
| `--input` | `-i` | yes | Path to input point cloud file |
| `--output` | `-o` | yes | Path to output file |
| `--epsilon` | | yes | DBSCAN epsilon radius (metres or whatever unit your coordinates use) |
| `--min-pts` | | yes | Minimum neighbor count for a core point |
| `--input-format` | | no | Override input format detection (`kitti`, `ply`, `csv`, `xyz`) |

## Format detection

Input and output formats are deduced from the file extension. If detection fails and no `--input-format` override is provided, the tool exits with an error listing the supported extensions.

`--input-format` only overrides **input** detection. Output format is always inferred from the output file's extension.

## Supported input formats

| Format | Extension | Notes |
| --- | --- | --- |
| KITTI binary | `.bin` | 4×f32 per point (x, y, z, intensity) — intensity is ignored |
| PLY | `.ply` | ASCII, binary little-endian, or binary big-endian; any numeric property type |
| CSV | `.csv` | Comma or whitespace separated; optional header row; columns: x, y, z |
| XYZ | `.xyz` | Same as CSV |

The PLY reader locates `x`, `y`, `z` properties by name regardless of their position in the property list. Other properties (normals, intensity, etc.) are skipped.

## Supported output formats

| Format | Extension | Notes |
| --- | --- | --- |
| PLY | `.ply` | Binary little-endian; per-point label (int) and RGB color (uchar) |
| CSV | `.csv` | Columns: x, y, z, label (-1 = noise) |

### PLY output schema

```
element vertex N
property float x
property float y
property float z
property int label
property uchar red
property uchar green
property uchar blue
```

`label` is 1-based for cluster members and `-1` for noise. `red`, `green`, `blue` are derived from the label using the golden-ratio hue method (see below).

## Label-to-color mapping

Cluster colors are generated by stepping through hues using the golden-ratio conjugate (≈ 0.618), which maximally separates adjacent cluster IDs in perceptual hue space:

- Noise → grey `[128, 128, 128]`
- Cluster *n* → HSV hue `(n × 0.618) mod 1.0`, saturation 0.8, value 0.9

Colors are stable and deterministic: the same label always produces the same RGB, regardless of cluster count or run order.

## Examples

```bash
# KITTI scan → PLY (format detected from extension)
vdbscan -i 000000.bin -o 000000.ply --epsilon 0.4 --min-pts 5

# ASCII PLY → labelled CSV
vdbscan -i scan.ply -o labels.csv --epsilon 0.5 --min-pts 3

# File with no extension → override input format
vdbscan -i velodyne_scan -o out.ply --epsilon 0.4 --min-pts 5 --input-format kitti

# XYZ point cloud
vdbscan -i cloud.xyz -o cloud_labelled.ply --epsilon 1.0 --min-pts 10
```

## Build

```bash
# debug
task build:cli

# release (recommended for actual use)
task build:cli:release
```

The binary is placed at `target/release/vdbscan` (or `target/debug/vdbscan` for debug builds).

## Parameter guidance

| Scenario | `--epsilon` | `--min-pts` |
| --- | --- | --- |
| Vehicle LiDAR (KITTI, ~64-beam) | 0.3–0.5 m | 5–10 |
| Dense terrestrial scan | 0.05–0.1 m | 10–20 |
| Sparse aerial LiDAR | 1.0–2.0 m | 3–5 |

Start with `epsilon` matching the typical point spacing in your densest expected cluster, and `min-pts` around 5. Increase `epsilon` to merge fragmented clusters; increase `min-pts` to suppress small spurious clusters.
